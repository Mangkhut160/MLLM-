# 电池审核项目设计方案

日期：2026-03-30

## 1. 项目背景

本项目基于现有课程资料、数据样本和训练代码，构建一套面向电动车/电池合规审核场景的多模态审核原型。项目目标不是实现一个泛化的内容审核平台，而是围绕单一业务场景完成从数据整理、模型训练、评测校准到推理决策的完整闭环。

当前业务输入由两部分组成：

- 用户填写的结构化信息：生产日期、电压、品牌、容量
- 用户上传的两张图片：品牌相关图片、参数/铭牌/充换电信息图片

系统需要根据文本字段和两张图片联合判断样本是否可以自动通过，或是否需要进入人工复核。

## 2. 项目目标

### 2.1 核心目标

- 构建一个可复现的电池审核原型项目
- 让模型输出结构化审核结果，而不是仅输出自由文本
- 支持“自动通过 / 人工复核”的线上决策逻辑
- 在第二阶段引入 GRPO，增强驳回标签、审核理由和结果一致性

### 2.2 业务目标

- 降低人工审核量
- 提高审核速度
- 在风险可控的前提下提升自动化通过能力

### 2.3 非目标

本阶段不包含以下能力：

- 企业级审核后台
- 完整人工审核工作台
- 账号权限、监控告警、数据库等服务化基础设施
- 全业务通用内容审核平台

## 3. 系统边界

### 3.1 输入边界

项目当前固定输入为：

- 表单字段：生产日期、电压、品牌、容量
- 图片 1：品牌相关图片
- 图片 2：参数/铭牌/充换电信息图片

### 3.2 输出边界

模型输出必须为结构化结果，至少包含：

- 品牌检测结果
- 日期检测结果
- 电压/容量检测结果
- 规格合规检测结果
- 图片规范检测结果
- 最终审核结果
- 驳回标签列表
- 可选的简短审核理由

### 3.3 决策边界

原型阶段不做自动驳回，只做：

- 自动通过
- 人工复核

系统只在关键审核点全部通过且置信度达标时自动通过，否则统一进入人工复核。

## 4. 标准数据源

本项目标准数据源固定为三部分：

- `data.csv`
- `brand_new/brand_new/*.png`
- `charge_new/charge_new/*.png`

其中：

- `data.csv` 是标准事实表
- 两个图片目录是标准图像资产
- 后续所有训练、评测、推理格式都从标准数据源派生

以下文件不纳入主流程：

- `config.json`

该文件不作为审核项目的标准数据源，也不作为主训练配置。

## 5. 数据语义定义

### 5.1 `data.csv` 字段定义

- `id`：样本主键，同时作为图片关联键
- `person_info`：结构化字段，当前格式为 `生产日期$%$电压$%$品牌$%$容量`
- `label`：人工审核标签结果，多个标签之间用 `|||` 分隔

### 5.2 当前数据特征

当前目录内的样本为偏驳回的小规模教学/原型数据，不代表真实线上分布：

- 大多数样本带有驳回标签
- 仅少量样本为空标签
- 不能直接代表完整的自动通过分布

该限制需要在后续评测和阈值校准中单独考虑。

## 6. 规范化样本对象

项目内部不直接使用原始 CSV 行作为训练输入，而是统一转换为规范化样本对象：

```json
{
  "sample_id": "460790679",
  "input": {
    "form": {
      "production_date": "2024-12-20 15:51:43",
      "voltage": "45V",
      "brand": "天能",
      "capacity": "72Ah"
    },
    "images": {
      "brand_image": "brand_new/brand_new/460790679.png",
      "spec_image": "charge_new/charge_new/460790679.png"
    }
  },
  "output": {
    "point_results": {
      "brand_check": "通过",
      "date_check": "通过",
      "voltage_capacity_check": "驳回",
      "spec_check": "通过",
      "image_check": "通过"
    },
    "final_result": "人工复核",
    "reject_tags": [
      "电池额定电压/额定容量与填写信息不一致"
    ]
  }
}
```

## 7. 标签映射规则

原始业务标签需映射到固定审核点：

- `使用电池不符合规范` -> `spec_check = 驳回`
- `使用电池规格严重超标，不符合国标要求` -> `spec_check = 驳回`
- `电池品牌图片与填写品牌信息不一致` -> `brand_check = 驳回`
- `电池生产日期与填写信息不一致` -> `date_check = 驳回`
- `电池额定电压/额定容量与填写信息不一致` -> `voltage_capacity_check = 驳回`
- `非电池相关信息图片` -> `image_check = 驳回`
- `非规范电池信息图片` -> `image_check = 驳回`

未命中的审核点默认视为通过。

只要任一关键审核点为驳回，最终结果默认不自动通过，而是进入人工复核。

## 8. 数据清洗与切分策略

### 8.1 清洗规则

- 删除无法同时关联两张图片的样本
- 标记并剔除损坏图片、非法路径样本
- 对 `label` 为空的样本显式标记为全审核点通过
- 对 `person_info` 解析失败样本单独输出异常集
- 统一历史命名差异和目录层级差异

### 8.2 数据切分

项目至少切分为三类数据：

- `train`：训练集，可做重采样和平衡增强
- `val_calibration`：阈值校准集，尽量保持真实分布
- `test`：最终测试集，严格不参与调参

## 9. 系统架构

### 9.1 离线训练链路

离线训练链路包括：

1. 标准数据源读取
2. 样本清洗与规范化
3. Prompt 基线验证
4. 第一阶段 SFT/LoRA 训练
5. 第二阶段 GRPO 训练
6. 评测与阈值校准

### 9.2 在线审核链路

在线审核链路包括：

1. 用户提交表单字段和两张图片
2. 推理模块生成结构化审核结果
3. 决策层读取关键审核点和最终结果置信度
4. 满足阈值则自动通过
5. 否则进入人工复核
6. 人工复核结果回流，形成数据闭环

## 10. 推荐技术路线

项目采用如下主路线：

`Prompt -> SFT/LoRA -> GRPO -> 阈值校准`

### 10.1 路线解释

- Prompt：用于业务验证和冷启动样本构造
- SFT/LoRA：用于学习稳定的结构化审核结果
- GRPO：用于增强标签、结果和短理由的一致性
- 阈值校准：用于将模型输出转为安全的自动通过决策

## 11. 模型方案

### 11.1 主模型

- 主模型：`Qwen2.5-VL`
- 微调方式：`LoRA`

选择原因：

- 与现有资料和训练骨架天然对齐
- 适合多图 + 文本统一建模
- 训练成本可控
- 后续可自然扩展到 `think + answer` 形式

### 11.2 第一阶段：SFT 审核 MVP

第一阶段目标：

- 学会输出结构化审核 JSON
- 学会输出各审核点和最终结果
- 学会输出驳回标签

第一阶段不强制生成复杂解释文本，重点是结构化输出稳定性。

### 11.3 第二阶段：GRPO 增强

第二阶段目标：

- 强化结构化结果与标签的一致性
- 引入短理由输出
- 提升审核结果稳定性
- 在安全阈值约束下提升自动化覆盖率

## 12. 数据格式设计

### 12.1 Prompt 基线格式

Prompt 基线输入应统一组织为：

- 两张图片
- 结构化文本字段
- 统一审核指令

输出为结构化审核结果，用于评估任务可行性和构造冷启动样本。

### 12.2 SFT 格式

第一阶段 SFT 输出固定为结构化 `answer`：

```json
{
  "brand_check": "通过",
  "date_check": "通过",
  "voltage_capacity_check": "驳回",
  "spec_check": "通过",
  "image_check": "通过",
  "final_result": "人工复核",
  "reject_tags": [
    "电池额定电压/额定容量与填写信息不一致"
  ]
}
```

### 12.3 GRPO 格式

第二阶段引入：

```xml
<think>参数图与申报容量不符，需人工复核。</think>
<answer>{...结构化审核结果...}</answer>
```

约束如下：

- `think` 目标长度约 30 字
- 最长不超过 50 字
- `answer` 必须保持结构化

## 13. 评测体系

项目评测分为四层：

### 13.1 审核点级指标

- 各审核点精确率
- 各审核点召回率
- 各审核点 F1

### 13.2 最终结果级指标

- `final_result` 精确率
- `final_result` 召回率
- `final_result` F1

### 13.3 标签级指标

- 标签精确率
- 标签召回率
- 多标签集合匹配率
- Top-1 标签准确率

### 13.4 工程可用性指标

- 结构化输出解析成功率
- 非法 JSON 率
- 缺字段率
- 平均推理时延
- `think` 平均长度
- `think` 超长率

## 14. 决策策略与阈值设计

### 14.1 核心原则

主业务目标为：

- 优先保证自动通过精确率
- 在该前提下尽量提升自动通过覆盖率

### 14.2 阈值机制

采用双层阈值：

- 点级阈值：关键审核点各自的通过阈值
- 总阈值：最终结果的通过阈值

只有满足以下条件才允许自动通过：

- 所有关键审核点均预测为通过
- 各审核点通过置信度超过点级阈值
- 最终结果通过置信度超过总阈值
- 输出结构合法

否则统一进入人工复核。

### 14.3 原型阶段限制

由于当前数据明显偏驳回，无法直接支持真实线上自动通过阈值的最终校准。当前阈值校准仅用于 MVP 级演示和相对比较，正式阈值需要补充真实分布样本后重新定标。

## 15. 为什么原型阶段不做自动驳回

当前阶段不做自动驳回，原因如下：

- 数据分布偏驳回，不足以支撑可靠自动驳回
- 误驳回的业务代价较高
- 当前课程资料推荐的是“基审 + 人审兜底”逻辑
- 原型阶段更适合先证明“安全自动通过”能力

因此原型阶段机器决策仅有：

- 自动通过
- 人工复核

## 20. 风险与应对

### 风险 1：数据分布偏驳回

影响：阈值校准失真、模型过度保守

应对：明确 MVP 定位，后续补充真实分布通过样本

### 风险 2：样本与图片路径对齐问题

影响：训练集构造失败、样本缺失率高

应对：建立统一数据规范化脚本

### 风险 3：结构化输出不稳定

影响：评测和决策层无法稳定解析

应对：第一阶段优先强约束输出格式

### 风险 4：GRPO 冷启动样本质量不足

影响：强化学习方向跑偏

应对：少量高质量冷启动样本优先

### 风险 5：解释文本被错误用于决策

影响：系统不稳定

应对：明确规定在线决策只依赖结构化输出和候选概率
